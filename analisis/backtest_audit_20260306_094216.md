## BUGS ENCONTRADOS Y CORREGIDOS

### BUG 1 — Strategy 1 (Back Empate): xG null gate bloqueaba 216 triggers
**Severidad:** CRITICA
**Impacto:** 216 BT_ONLY → 6 BT_ONLY (mejora de 210 triggers)

**Causa raíz:**
```python
# ANTES: gate bloqueaba partidos sin cobertura xG (ligas menores)
if draw_ver != "off" and (... and
    xg_local is not None and xg_visitante is not None):
```
El BT `_filter_draw()` en optimize.py tiene un short-circuit bug que silenciosamente pasa bets con xG=None (la condición `xg is not None and xg >= p["xg_max"]` devuelve False cuando xg=None, sin rechazar el bet). El live detector, en cambio, rechazaba TODOS los partidos sin xG.

**Fix aplicado:**
```python
# AHORA: xG null tratado como "pasa el filtro" — idéntico a _filter_draw
if draw_ver != "off" and (... and
    goles_local == 0 and goles_visitante == 0):
    # Sin gate xG externo
    xg_total = (xg_local + xg_visitante) if (xg_local is not None and xg_visitante is not None) else None
    _xg_ok = _draw_xg_max >= 1.0 or xg_total is None or xg_total < _draw_xg_max
```

### BUG 2 — Strategy 4 (Goal Clustering): minuteMax hardcodeado a 80
**Severidad:** ALTA
**Impacto:** 120 BT_ONLY reducidos a 9 (después de ajuste del reconcile)

**Causa raíz:**
```python
# ANTES: hardcodeado <= 80
if ... and _cluster_min <= minuto <= 80:
    ...
    if row_min and 15 <= row_min <= 80:
```
El BT `CLUSTERING_PARAMS["v2"]["min_max"] = 90`, por lo que la comparación BT vs live era desigual. Además el reconcile necesitó aplicar el filtro de config (`minuteMax=60`) al lado BT para que la comparación fuera justa.

**Fix aplicado:**
```python
# AHORA: usa _clustering_min_max (default=90, overrideable por config)
_cluster_max_eff = _clustering_min_max
if ... and _cluster_min <= minuto <= _cluster_max_eff:
    ...
    if row_min and 15 <= row_min <= _cluster_max_eff:
```

**Nota adicional:** El reconcile (`aux/run_reconcile.py`) se actualizó para filtrar BT clustering por `config.minuteMax` antes de comparar, alineando la semántica con lo que cartera.ts aplica.

### BUG 3 — Strategy 7 (Tarde Asia): NO implementada en detect_betting_signals
**Severidad:** ALTA
**Impacto:** 100 BT_ONLY → 0 BT_ONLY (100% cobertura)

**Causa raíz:** La estrategia `tarde_asia` existía en el BT (`analyze_strategy_tarde_asia`) pero no tenía un bloque en `detect_betting_signals`. El sistema live nunca generaba señales tarde_asia.

**Fix aplicado:** Implementado nuevo bloque STRATEGY 7 en `detect_betting_signals`:
- Trigger: minuto <= 15 + cuota back_over25 disponible
- Detección de liga: replicada de `analyze_strategy_tarde_asia` (URL + nombres de equipos)
- Fix adicional: normalización de guiones→espacios para detectar correctamente nombres como `al-hilal` → `al hilal`
- Versión activada via `versions.get("tarde_asia", versions.get("tardeAsia", "off"))` — compatible con analytics.py que ya pasa esta clave

## DISCREPANCIAS ESTRUCTURALES (no corregibles sin cambiar comportamiento live)

### 1. Odds Drift — 41 BT_ONLY
**Causa:** El BT dispara en la PRIMERA fila donde se cumple la condición. El live requiere `stability=2` (señal presente en 2 filas consecutivas). Muchos triggers drift son efímeros (1 sola fila), pasando el threshold en un momento y desapareciendo en la siguiente captura.

**Ejemplo real:** `kagoshima-utd-fc-ryukyu` — drift dispara en row 25 (min=13) con drift=159.4%, pero en row 26 (min=14) la señal desaparece. BT cuenta el bet; live no.

**Decisión:** Mantener como está. El stability=2 del live es CORRECTO y más conservador (evita falsas señales transitorias). La diferencia es intencional.

### 2. Pressure Cooker — 28 BT_ONLY
**Causa:** El BT `analyze_strategy_pressure_cooker` confirma el score con `abs(check_min - actual_min) <= 3`, que incluye FILAS FUTURAS al trigger (look-ahead). El live solo mira las últimas 6 filas pasadas. Cuando un gol ocurre justo en el minuto 65 (trigger), el BT cuenta filas a min 65+3=68 como "confirmación", mientras live solo tiene la fila del trigger.

**Ejemplo:** `kf-drita-celje` — gol a min 74 establece 2-2, BT cuenta la confirmación incluyendo min 77 y 79 (futuras). Live a min 74 solo tiene 1 fila con score 2-2.

**Decisión:** No corregible. El BT tiene look-ahead bias aquí. En live esta es la diferencia esperada.

### 3. LIVE_ONLY draw (45 casos)
**Causa:** BT `DRAW_PARAMS["v2r"]["poss_max"] = 20` (hardcoded en optimize.py), mientras config tiene `draw.possMax = 25`. El live usa el valor del config (25), lo que permite bets con poss_diff entre 20-25 que BT rechaza.

**Decisión:** No es un bug live — el config del usuario tiene possMax=25. El BT filter usa valores hardcoded en optimize.py que difieren del config. Para eliminar los LIVE_ONLY, habría que alinear optimize.py con el config (pero optimize.py no debe modificarse).
