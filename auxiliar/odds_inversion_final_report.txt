EXECUTIVE SUMMARY
================================================================================

The initial analysis flagged 100% of en_juego matches (433/433) as having
"anomalies" using a simple threshold check (winning team back odds < 2.0).
After correcting for false positives, the real anomaly rate is:

  Genuine inversions (3+ rows):     29/1404 matches = 2.1%
  Sustained inversions (8+ rows):   11/1404 matches = 0.8%

VERDICT: RARE NOISE, not a systematic scraper bug.
The 100% false-positive rate in the initial scan was due to 3 data artifacts:
  1. Stale/suspended market rows (both odds near 1.0, NaN, or identical values)
  2. Score-transition rows (1-2 row glitch at the exact moment a goal is scored)
  3. Big-favorite legitimate markets (Benfica at 1.66 while trailing 0-1 to Arouca)

================================================================================
1. WHY THE INITIAL SCAN WAS MISLEADING
================================================================================

The initial script used this threshold:
  "home_leading AND back_away < 2.0" → SEVERE anomaly

This catches 1,378 "severe" rows across 418 matches.
But 1,014 of those 1,378 rows (69%) are NOT real inversions:

  Stale market (both odds < 1.1):    247 rows  — market suspended, no liquidity
  NaN values:                        195 rows  — scraper captured nothing
  Same value (|bh - ba| < 0.05):      93 rows  — stale/placeholder data
  Marginal cases (genuine close game): 479 rows — legitimate market behavior

  Real inversions (bh/ba truly swapped): 559 rows across 158 matches
  But most of those 158 matches have only 1-2 rows (score transition glitches)

================================================================================
2. THREE TYPES OF "FALSE POSITIVE" ANOMALIES
================================================================================

TYPE A: STALE MARKET ROWS
  Pattern: back_home = 1.02, back_away = 1.02 (or both near 1.0)
  Cause: Betfair market suspended (half-time, goal scored, VAR check)
  Example: al-hilal at min=8, score=1-0, bh=1.02, ba=1.02
  Impact: 247 rows. These rows are useless data regardless.

TYPE B: SCORE TRANSITION GLITCH (1-row race condition)
  Pattern: At the exact minute of a goal, one data column updates before another.
  Example: Lincoln-Northampton min=10: home just scored (1-0),
    back_home = 3.75 (jumped to wrong value), back_away = 1.34
    Next row (min=11): immediately corrects to back_home=1.42, back_away=9.00
  Cause: Scraper reads odds page and score page with slight timing difference.
  Impact: Very common (~200+ isolated 1-row cases). Not a structural problem.
  Effect on strategies: Negligible. Strategies require multiple confirmation rows.

TYPE C: BIG FAVORITE LEGITIMATE MARKET
  Pattern: Heavy pre-game favorite (e.g. back_away = 1.29) concedes early goal.
  Market still prices them as favorite at 1.66 while trailing 0-1.
  Example: Arouca (HOME, 11.5 pre-game) vs Benfica (AWAY, 1.29 pre-game)
    Score 1-0 Arouca, but back_away = 1.66 (Benfica still being backed heavily)
    Arouca back_home = 3.1 to 5.6 range after scoring
  Analysis: This is NOT a column swap. It IS legitimate Betfair market behavior.
    If it were a swap: back_home would need to be 1.66 (Arouca as 1.66 favorite?)
    That doesn't make sense for an 11.5 pre-game underdog.
    Benfica bettors keep backing them after an early concession — real market.
  Impact: Generates many "severe" flags but does NOT represent bad data.

================================================================================
3. THE REAL ANOMALIES
================================================================================

After excluding the 3 false-positive categories, genuine inversions are:

  Rows where winning team's odds ROSE ABOVE their pre-game odds level:
  (Winning should cause odds to DROP, not RISE — this is the true anomaly)

  Matches with 3+ genuine rows:  29 (2.1% of matches with live data)
  Matches with 8+ genuine rows:  11 (0.8% of matches with live data)

GENUINE INVERSION PATTERNS:

A) FROZEN SCORE IN CSV
   Example: Seattle Sounders vs Colorado (partido_seattle-sounders-colorado-apuestas-35189512.csv)
   Score shows 1-0 from min 16 all the way to min 90 (CSV score frozen)
   But odds rise to 220-280 for back_home (market says home is extremely unlikely to win)
   Reality: Score must have changed (Colorado equalized or led) but scraper didn't capture it
   The ODDS are correct; the SCORE in CSV is wrong (frozen at old value)
   Impact on strategies: Score-based strategies would use wrong score → false signals possible

B) GENUINE ODDS COLUMN SWAP
   Example: America de Cali vs Santa Fe (partido_america-de-cali-sa-santa-fe-apuestas-35238051.csv)
   Shows alternating rows: some with back_home=6.80, back_away=1.11 (fixed ghost values)
   Then normal rows, then ghost rows again
   Pre-game: back_home=2.0, back_away=4.2 (America Cali was the favorite)
   The 6.80/1.11 pattern appears to be a scraper capture failure (wrong element grabbed)
   Only 1 match with this exact ghost pattern identified.

C) SCORE OSCILLATION
   Some matches show score going 0-0 → 1-0 → 2-0 → 1-0 (drops back)
   This is the known "oscillating scores" data quality issue (documented in CLAUDE.md)
   When score goes from 2-0 back to 1-0, a "winning while high odds" condition can trigger
   for the 2-goal row without being an actual inversion.

5. IMPACT ON BETTING STRATEGIES
================================================================================

The 32 strategies in csv_reader.py are gated by SCORE conditions first.
Strategies like away_fav_leading require: goles_visitante > goles_local (away LEADING)
So a "big favorite while trailing" case (TYPE C above) would NOT trigger those strategies
because the score condition fails (they're not actually leading).

RISK AREAS:

1. FROZEN SCORE MATCHES (1 confirmed: Seattle-Colorado)
   Score stuck at 1-0 while real score diverged → strategies reading wrong score
   Could produce false bet signals if the strategy only looks at score.
   Action: Low priority (only 1 match identified with this pattern)

2. STALE MARKET ROWS (247+ rows)
   Odds near 1.0 during market suspension → strategies using odds thresholds
   Could produce false triggers for strategies with very low odds thresholds (<1.2)
   Action: Already partially handled by null-check in most strategies.

3. SCORE OSCILLATION ROWS (existing known issue, documented in CLAUDE.md)
   Score flickers 2-0 → 1-0 → 2-0 during live data capture
   Already documented. Not an odds inversion problem.

STRATEGIES MOST AT RISK:
  - away_fav_leading: uses back_away as threshold → protected by score gate
  - home_fav_leading: same protection
  - back_draw_00: uses draw odds only → not affected by home/away odds
  - lay_draw_away_leading: uses score + odds → protected by score gate
  - cs_close: uses back_home/back_away proximity → could trigger on stale 1.02/1.02 rows
    but requires specific score too (protected)

CONCLUSION: The odds inversion issue does NOT require code changes.
The data artifacts are:
  - Rare (2.1% genuine, 0.8% sustained)
  - Already partially filtered by the existing score-gating in strategies
  - Consistent with the documented "null data gap" behavior (CLAUDE.md)

6. RECOMMENDATIONS
================================================================================

1. NO CODE CHANGE NEEDED for the betting strategies.
   The score-first gating in all 32 triggers provides sufficient protection.

2. OPTIONAL: Add an odds-sanity check helper that flags rows where:
   (back_home < 1.05 AND back_away < 1.05) → market suspended, skip row
   This would clean up the 247 stale market rows.

3. MONITOR: The 29 matches with genuine inversions (2.1%) should be cross-checked
   against placed_bets.csv to see if any of those matches produced actual bets
   during the anomalous periods.

4. DATA QUALITY: The Seattle-Colorado frozen-score case indicates that score data
   can occasionally freeze in CSV files while odds continue to update.
   This is a scraper capture issue, not an odds column swap.
